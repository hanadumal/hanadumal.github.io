生活在这样一个信息爆炸的时代，我们很容易陷入到知识恐慌中，我们担心世界进步太快，担心自己成长太慢，担心错误太多的美好。所以经常不自觉的会陷入到·蝜蝂者·困境中，于是乎我们不断地买买买书，不断的听听听课，不断的熬夜失眠或学习或刷剧。经常听到在北京的人说，充足舒适的睡眠很奢侈；第二天继续自己的焦虑或者慌张，我们在陷入了困境中不能自拔，无法自救。即使有人帮助我们把负重拿走，或是自我觉醒决定断舍离了。用不了多久大抵还是会重蹈覆辙。

So?到底是哪里出了问题？为什么我的知乎收藏夹越来越满？为什么我的浏览器的Bookmark文件夹越来越大？为什么我的稍后阅读成为了再也不读？为什么我在Kindle上看书的速度永远赶不上买书的速度？

首先回答为什么会焦虑？

  • 此刻我已经快30岁了，刚刚开始职业生涯4年，如果程序员只能干到35，那离退休不远了。如果能干到50退休，那么还有20年的职业时间。

  • 在这个行业里，身边的同事已经有96 97年生人了。学习速度比我快，加班能力比我强，要的钱比我少，我用什么去和他们竞争。假设我对物质条件要求不高，覆盖在北京的基础开销之上略微有结余即可，那么房贷的压力也是继续会要求我继续工作。

  • 你作为高中的尖子生、985硕士毕业生，面对形形色色财务自由的人群，面对同龄人中家境优渥的选手，对比产生的嫉妒如何磨平？

  • 对于已经失去了的选择的后悔

  • 没有成就感的工作，会让你觉得在浪费自己宝贵的时间。越是长大越是发觉世间最宝贵的资源乃是人的时间。你甚至没有权利选择不把它花掉。过期不候强制消费。当你把时间用在无法获取成就感，无法获取乐趣的事情上面时。迷茫和痛苦倒闭你忧虑恐慌成为蝜蝂。

那么如何才能不恐慌？

  • 技术其实是很有趣的事情。真正的有机会受教育其实是在18岁以后，18岁以前接受的更像是灌输，而不是教育。细细想来在18岁的每年里你人生体验的密度是否足够大，对于你目前的焦虑和迷茫程度有很大的关联。18～28岁的10年时间里，浪费了到底多少时间，如果每天2小时，2 * 3650 = 7300小时，如果按照一万小时定律距离成为专家也不需要太久了。然而现实是你并没有在任何一个领域成为专家。你没有成为一个当下火热的短视频直播领域的大V；没有成为某个技术的专家；没有成为理财届的银行螺丝钉。纵观你们这一代人的群体，比你优秀的似乎总是那么多。1990年出生人口2200W，如果你刚好是前10%的位置，那么中国有200W人同龄人比你要强，更遑论88～91年都可以称之为你的同龄人，这样看来有800W同龄人都比你强大，资源比你更丰富。额外加之改革开放以来家庭财富快速积累的家庭中的二代。总之有比自己更强大，更有钱，更有趣的人很多很多。

  • 希望你能在自己每周五天，远超40小时的工作时间里获得乐趣和成就感。目前在熟人手下干活，其实应该有更多的属于自己成长的时间，不想干的工作都可以分派给其他人，业务成果的好坏对自己来说并不是最重要的，在自己32岁的时候，成为某个开源项目的PMC，成为某个项目的专家才是目前最迫切的需求。

  • 所谓的财务自由并不是真的不工作。

做哪些事在能让自己的信息，让自己的生活回到正轨？

  • 互联网上成PB的信息都可以提供出来，作为一个人不可能学习完，了解完所有的知识，要有筛选，要把信息转换成为知识，要进行信息的压缩，要构建知识的框架。要找到好的信息输入源头。

  • DRY原则，要进行抽象，把工作中遇到的情况，抽象成可复用的代码，不断积累自己可应对的业务场景。每日维护更新这个代码库，这是作为计算机软件工程师的立足的根基。

  • 要说跟新人有什么不同。就是你在这浑浑噩噩的十年中，不断挣扎，不断尝试，不断自我怀疑得到的一些结论。找到的哪怕一点点的星光。

  • 让自己成为一个对别人有吸引力的人。让自己成为一个健身大达人，拥有强壮的体魄，拥有清醒的精神状态。让自己能做的事情更多，让自己掌握工业界语言Java。让自己明年可以不用中文表达这些想法的碎片。

## OS 
1. select poll epoll
- 打开的文件描述符数量有限制，epoll只跟系统内存有关系，select, poll有限制，默认1024,当然可以修改以后，重新编译OS内核。
- 但是select poll是轮训所有的fd, 有数据就处理，却又就跳过，所以把fd改大了，效率就下降了。epoll只处理就绪的常fd, 它有一个就绪的队列，每次只查询该队列的数据，然后处理。另外select、poll需要把数据从内核态拷贝到用户态，但是epoll使用了mmap的机制省略了这个步骤。

# reduceByKey 和 groupByKey的区别是什么？
reduceByKey是对每个key的value进行函数自定义函数进行merge,它可以现在本地就先merge。

groupByKey是对每个key生成一个sequence,那么也就是说reduceByKey等于groupByKey+map操作。在很多key-value对的情况下，reduceByKey优于groupByKey

做过的项目

  1. 数据调度平台
    主从结构的任务分发平台，包括Web前后端模块、任务提交执行模块。用户可以在平台上编写SQL任务、脚本任务。在Web平台上任务进行管理：编辑内容、设置例行时间、选择执行机器、结果推送等。当任务到了规定的时间点，则由执行机进行调度执行。根据User、Group、Role来确定Permission提交执行。主要的技术框架Django、MySQL、thrift。同时在平台上还集成了机器管理、前端统计点位管理、仓库元数据信息管理的功能。

  2. 数据同步平台

  3. 实时计算平台
    实时计算平台主要是为了服务实时统计，比如说每年百度高考报志愿活动的人数统计。架构就是Nginx -> Flume -> Kafka -> Strom -> MySQL。这部分工作主要在于Flume的sinker的定制和Storm应用的开发上。从日志落盘开始到计算完成一共大概需要2s时间。

建堆的过程

  1. 当前节点的，左右子节点，分别为2i+1, 2i + 2
  2. (arr, n, i) arr为输入数组，n为数组大小，i为当前节点的index
  3. 先比较i, 2i + 1节点的大小, 获取largest
  4. 再比较largest和2i + 2节点的大小
  5. 如果largest不等于i, 那么交换largest和i, 并且针对largest节点继续进行heapify

查找的过程

  1. 从n-1至0], 也就是从底向顶，建立堆。
  2. i 从n-1到0), 交换0和i的数据，恢复节点0的堆。
  3. 结果为增序

bin heap的定义

  1. bin heap是一个完全二叉树，或近似完全二叉树
  2. 父节点的value大于（小于)子节点的value
  3. 每个节点的左右子树都是一个二叉堆（因为这条，所以建立堆的时候需要从bottom到top建立)

堆排序的时间复杂度，为什么比快排慢？

时间复杂度为O(nlog), heapify的时间复杂度为O(logn), 建立堆的复杂度为O(nlogn), 排序的时候需要n-1次循环，所以复杂度O(nlog), 整体复杂度O(nlogn)

堆排序需要将A[0]与A[n-1]进行交换，将A[0]重新下沉到合适的位置。每次新交换上来的A[0]十分小，能够呆在A[0]不需要交换的概率很低，需要经过多次比较和交换才能到达合适的位置。更好的办法是，每次只有1/2的概率需要移动。堆排序和快排虽然都是O(nlog)的时间复杂度，但是堆排序的常数更大。

  1. 注册自己的域名
    注册域名需要像域名商的网站商发起购买。在它的网站上筛选到喜欢的域名如果尚未出售就可以直接购买。如果已经售出那么除非你真的志在必得，否则还是推荐换一个。那么我们要在哪个域名商店铺里进行购买呢？类比于网购我们可以去淘宝、京东、拼多多，域名商店也有很多。hanadumal.com我是在namesilo上购买的。我使用过的域名购买有阿里云、godaddy、namesilo。国内的暂且不表了。godaddy这家伙是在是坑，虽然是我买过最多的一家，但是现在已经完全不会考虑从godaddy再买了。首先价格很高，第一年通常会有优惠，但是如果你一次购买大于一年，后面的年费比其他家高很多。举例来说，hanadumal.com我在godaddy上查询，发现购买价格是470+。续费价格更贵，在godaddy一上车购买一年试试水，你会发现第二年快到期续费的时候居然比一开始买两年的第二年价格高很多。隐私保护还要额外收费60元/年。namesilo上我买的第一年6.99到，续费8.99刀/年。即使按照人民币汇率7来简单估算，5年都按照8.99刀，折合人民币也只要315元不到。另外namecheap也可以，价格比namesilo稍微贵一点，但是比godaddy良心的多。namecheap和namesilo都提供免费的隐私保护（永久)。我这里打开namecheap其实比namesilo快很多，感觉商店UI也更亲人。如果不在乎稍微多花钱namecheap也是不错的选择，如果第一关心的要点是性价比，那直接namesilo就对了。

FAQ

  1. hexo server -p 8080 后访问http://localhost:8080不到页面?

  2. scaffolds有什么用?
    scaffolds是脚手架，或者说是创建posts时使用的模版。当执行了hexo new photo ‘My Gallery’以后，会在scaffolds目录下寻找
    photo.md作为创建source/_posts/My-Gallery.md的模版。

  3. deploy到github以后，通过.github.io访问点击进入文章详情，提示404。
    我遇到过的情况是，浏览器本地缓存导致的，清空缓存或者换个浏览器可以正常访问。部署到github以后，用.github.io来访问一般不会出现问题。

  4. 绑定域名DNS解析A记录、CNAME以后，通过新绑定的域名访问，提示404。
    一个有可能的原因是，project的_config.yml修改了url配置, 修改后重新发布，访问正常。
